## `$ whoami --dsa`

This repository is my **engineering journal** for Data Structures and Algorithms.

Every file here represents a problem understood — not just solved. I document my approach, complexity analysis, and key learnings alongside every solution. Over time, this repository becomes a map of how my problem-solving thinking has evolved.

**Why this repository exists:**

- DSA fundamentals are the language engineers use to think about performance, trade-offs, and correctness. Knowing frameworks is not enough.
- Consistency compounds. A problem a day, documented clearly, builds pattern recognition that no tutorial can replicate.
- I want every recruiter, engineer, or student who opens this to immediately understand: this developer practices deliberately, not casually.

**How it is organized:**

Every topic lives in its own directory. Every solution includes a header comment with problem source, difficulty, approach, and complexity. The `Notes/` directory contains cross-topic insights and pattern summaries I've written in my own words.

## `$ cat dp_hub.md` 🔥

Dynamic Programming is my current primary focus. This section is my DP knowledge base, updated as I learn.

<details open>
<summary><strong>DP Fundamentals</strong></summary>

<br/>

| Concept | Description | My Status |
|:--------|:------------|:---------:|
| **Memoization** | Top-down recursion + cache. Write the recursion first, then add the cache. | 🟡 Learning |
| **Tabulation** | Bottom-up iteration. Fill a table from base cases. Usually faster in practice. | 🟡 Learning |
| **State Definition** | The hardest part. Ask: "What information do I need at each step?" | 🟡 Learning |
| **Transition Logic** | How does `dp[i]` depend on `dp[i-1]` or `dp[i-k]`? | 🟡 Learning |
| **Base Cases** | What is the answer for the smallest possible input? | 🟢 Clear |

</details>

<details>
<summary><strong>DP Patterns — My Learning Tracker</strong></summary>

<br/>

| Pattern | Example Problems | My Notes |
|:--------|:-----------------|:---------|
| 1D DP | Climbing Stairs, House Robber, Coin Change | Classic recurrence |
| 2D DP | Unique Paths, Edit Distance, Longest Common Subsequence | Table filling |
| 0/1 Knapsack | Subset Sum, Target Sum, Partition Equal Subset | Include or exclude |
| Unbounded Knapsack | Coin Change II, Rod Cutting | Item can be reused |
| LIS (Longest Increasing Subsequence) | LIS, Russian Doll Envelopes | O(n log n) with patience sort |
| LCS (Longest Common Subsequence) | LCS, Shortest Common Supersequence | 2D table, string comparison |
| DP on Trees | Diameter, Max Path Sum, House Robber III | Post-order DFS |
| DP on Graphs | Longest Path in DAG | Topological sort + DP |
| Bitmask DP | Traveling Salesman, Assignment Problems | State = visited set |
| Interval DP | Matrix Chain Multiplication, Burst Balloons | Range [i..j] |

</details>

**My DP approach for every new problem:**
1. Recognize it's DP (optimization or counting)
2. Define the state
3. Write the recurrence relation
4. Identify base cases
5. Code top-down (memoization) first
6. Convert to bottom-up (tabulation) if needed
7. Optimize space if possible

## `$ cat interview_strategy.md`

My approach to using this repository for interview preparation:

**Pattern recognition over memorization.** Every time I solve a problem, I ask: "What is the core pattern here?" Memorizing solutions fails under interview pressure. Recognizing patterns does not.

**The 4-step interview approach I practice:**
```
1. Clarify   → Understand constraints. Ask about edge cases before coding.
2. Plan      → State the approach out loud. Complexity analysis before code.
3. Implement → Write clean code. Name variables clearly.
4. Test      → Trace through examples. Check edge cases.
```

**FAANG Preparation Targets:**

| Company | Primary Focus | Secondary Focus |
|:--------|:-------------|:----------------|
| Google | Arrays, Strings, Graphs, DP | System Design |
| Microsoft | Trees, Graphs, OOP | API Design |
| Amazon | Arrays, Strings, Sorting | Leadership Principles |
| Meta | Graphs, Trees, Arrays | Recursion |
| General Backend | All + System Design | SQL, Concurrency |

**Consistency strategy:** One focused session daily. Three problems minimum. One hard problem per week. Weekly revision of patterns I got wrong.

## `$ cat CONTRIBUTING.md`

This repository is primarily my personal learning journal, but contributions that improve the quality for other learners are welcome.

**What I welcome:**

- Improving an explanation in a solution's header comment
- Adding an alternative approach with a different time/space trade-off
- Fixing a bug in a solution
- Improving documentation clarity in `Notes/`
- Suggesting a better folder structure with reasoning

**What I don't want:**

- Adding raw solutions without explanation headers
- Changing working code without a clear reason
- Adding problems already covered without meaningful improvement

**How to contribute:**

```bash
# 1. Fork the repository
# 2. Create a branch
git checkout -b improve/topic-name

# 3. Make your changes
# 4. Commit with a clear message
git commit -m "docs: improve explanation for sliding window pattern"

# 5. Open a Pull Request with a clear description
```

**PR description template:**

```
What I changed:
Why it's better:
Tested on: (problem links)
```
